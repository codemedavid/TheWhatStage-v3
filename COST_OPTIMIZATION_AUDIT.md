# WhatStage Cost-Optimization Audit

> **Status: RECOMMENDATION DOCUMENT. Nothing in this report has been implemented.** All numbers are estimates anchored to the owner's $1.58 OpenRouter sample (DeepSeek V4 Flash: 9.53M tok / 2,440 req / $0.95; embeds 1,330 req; Cohere rerank ~$0.20). Treat $ figures as order-of-magnitude.
>
> _Produced by a 12-agent audit (8 cost-center profilers + 3 cross-cutting analyses + synthesis), 2026-06-02._

---

## 1. TL;DR

- **What makes the bill float:** ~90% of spend is DeepSeek-Flash **input tokens**, and the dominant driver is the per-message combined reply+classify call (`classify.ts:237`) at **~3,900 prompt tokens fired on every inbound message**, of which **~2,400 tok (60-80%) is a byte-identical static prefix re-billed at full price every turn** because no prompt cache is engaged (`llm.ts:79-87` sends no `cache_control`).
- **The single biggest lever:** **Engage provider-side prompt caching on the static persona/rules/stage-instruction prefix.** Identical tokens, cheaper billing, zero quality risk. Worth ~**30-40% of the total bill** on its own and scales linearly with message volume.
- **A 30% leak that this codebase does not even generate:** GPT-4o-mini (621K tok / 187 req) and DeepSeek V4 Pro (124K tok / $0.33) have **zero source in the repo** (exhaustive grep). They are account-level (shared key, stale deploy, or OpenRouter auto-routing). ~$0.45-0.50 of $1.58 (~30%) is killable at the OpenRouter account level **today** with no code change.
- **The "no-op classifier" tax:** `RAG_CLASSIFIER_MODEL == RAG_LLM_MODEL == deepseek/deepseek-v4-flash` (`.env.local:17,26`), so `model-router.ts` and every "cheap classifier" path saves **$0**. The cheap-tier abstraction exists end-to-end but is wired to the expensive model.
- **Obvious waste to delete:** follow-up/reminder sequences generate 7 messages as **7 separate LLM calls** (and the reminder path regenerates them **again** at fire time = ~14 calls where 1 batched call would do); deep-reclassify re-dumps **60 raw messages** to redo a stage classification the main call already produced.
- **Projected before→after at scale:** stacking the safe wins cuts the dominant per-inbound amortized token count from **~4,750 → ~2,000-2,200 flash tok/msg (~55%)**. Combined with killing the off-repo leak, **realistic total-bill reduction is ~55-65%**, and the cost curve flattens as clients onboard.

**Why ~3,900 tokens/request** (`classify.ts:237`): 9.53M tok ÷ 2,440 req. The prompt is `persona+groundrules (~900 tok) + static stageInstruction prose (~1,500 tok) + RAG context (up to 6 chunks × 1,024 tok) + leadContext + media block + 12-turn history + summary`. The static instruction prose alone (`prompt-builder.ts:135-219` stable block ~1,485 tok + `classify.ts:559-712` stageInstruction ~2,933 tok) is **~4,400-4,800 byte-identical tok on an action-page turn — already exceeding the 3,900 average**, meaning static instructions dominate and the volatile user data is the minority.

**Why 1,330 embed requests** (exceeds 1-per-message): (a) the **media selector embeds a second string** (`customerMessage+refText`, `selector.ts:85`) when a folder ref is present instead of reusing the retriever vector (`retriever.ts:53`); (b) `recommend_product`/`recommend_property` each add an embed (`recommend.ts:109`, `recommend-property.ts:114`); (c) CRAG rewrite adds one on zero-candidate searches (`retriever.ts:168`); (d) ingest `embedBatch` requests count too. **No embedding cache exists anywhere** (`openrouter-client.ts:28-63` — no LRU/memo/Redis).

## 5. The follow-up "1-click batch generate" question

**Verdict: YES — the owner's instinct is correct.** This is real, identifiable waste.

**Current state (the waste):**
- Follow-up engine: 7 touchpoints = **7 separate LLM calls** (`generateMessage.ts:132`, lazy one-per-fire). With DEFAULT settings every slot has an instruction, so even slot 0 (designed to use the free curated pool) pays for a call (`settings.ts:63`).
- Reminder-sequence path is **worse**: it generates all 7 at seed (`sequence-seed.ts:77` Promise.allSettled) **AND regenerates each at fire** (`fire.ts:175`, with `pre_generated_text` demoted to fallback `fire.ts:184`) = **~14 generations where 7 (or 1 batched) would do**.
- The shared ~250-500 tok preamble (rules + personality + Manila time) is re-sent on every call. Token math: per-call generic ~360 prompt → 7 calls ~2,520 prompt; **batched ~460 prompt** → **~5.5x fewer prompt tokens**.

**How much it saves:** ~3.5x fewer total tokens per sequence (~2,800 → ~740 tok), ~40% off that feature, ~3-5% of the total bill. Small in absolute $ today, but it is pure waste and the per-sequence curve grows with lead volume.

**Freshness / quality tradeoff:** the honest catch is that batch-at-seed loses the ability to reference messages exchanged between touchpoints. But: (a) **generic-kind leads** (`gate.inboundCount < 4`, `seed.ts:57`) have **zero transcript dependency** — fully batchable with no freshness loss; (b) follow-up sequences exist precisely **because the lead went silent**, so the common case has no new messages anyway; (c) if the lead replies, the schedule is cancelled and re-seeded (`seed.ts:42`), so stale generic drafts are discarded naturally.

**Recommended hybrid design (analysis only):**
1. **Generic-kind leads:** batch-generate all enabled touchpoints in **one `json_object` call at seed** (all instructions/attachment hints known from `offsets_snapshot`, `settings.ts:73-96`). Persist drafts; `fire.ts` just pops the pre-generated text. Collapses up to 7 calls → 1.
2. **Real-kind leads:** batch at seed using the seed-time transcript, **plus a watermark** — store `last_inbound_at` at generation. On each fire, cheaply compare `messenger_messages.created_at` to the watermark (no LLM). Nothing new → send the pre-generated draft (**0 LLM calls**). New messages → regenerate **only that one touchpoint** lazily.
3. **Both:** keep the per-slot curated fallback pool (`sequence-fallbacks.ts`) + a 1-retry for parse failures (one bad JSON array must not lose all 7).
4. **Stop fire-time regeneration** on the reminder path (item #3 / #10 above) — this alone halves that path independent of batching.

## 6. Messaging (chatbot reply) optimization — the plan

1. **Combine classify+answer: ALREADY DONE — do not re-engineer.** `answerWithClassification` (`classify.ts:237`) emits reply + stage_change + action_page + recommend_* in **one JSON envelope**. `answer()` (`route.ts:602`) is a fallback only when the combined reply is empty; the in-function fallback (`classify.ts:320`) fires only on parse failure. The "halve input by combining calls" lever is already captured. The remaining duplicate-reasoning calls are decideForceSend (`force-send.ts:161,179`) and deep-reclassify — fold/gate those (items #15, #11).
2. **Prompt-cache the persona/RAG prefix (highest-$ lever).** Make the prefix contiguous (item #9), then engage caching (item #8). The ~2,400-4,800 tok static prefix billed at a ~75-90% discount cuts ~45% of input on the dominant call.
3. **Gate deep-reclassify (item #11) and keep summarize as-is.** Summarize is already well-gated (`shouldRollSummary`, interval 8, fire-and-forget `route.ts:705-717`) — only minor win in widening 8→12. Deep-reclassify is the real target: signal-gate it and feed the summary instead of 60 raw msgs.
4. **Prune context (item #2).** Static prose ~50% down; maxContext 6→3; chunkMaxTokens 1024→512. ~1,100-2,500 tok/turn off input. Keep the IRON RULE + LINK_TEASE regex backstop (`classify.ts:927`).
5. **Cheaper classifier model (item #4) — but NOT for the main reply.** The main envelope writes customer-facing Taglish sales copy; keep it on Flash. Move only deep-reclassify / classifyOnly / reminders / comments-classify to a genuinely cheaper tier by un-aliasing `RAG_CLASSIFIER_MODEL`.
6. **Keep `response_format=json_object`.** Do **not** switch to tool/function-calling — it would ADD a tool-definition payload (~100-300 tok/turn) and reintroduce the tool-call-artifact leakage the code spends ~60 lines stripping (`sanitizeReply`, `classify.ts:857-920`). The only structured-output saving is to shrink the inline schema/anti-leak prose, bundled into item #2.

## 7. Auto-sort optimization

The KB-driven **stage-suggester** (`stage-suggester.ts:162`) is low-frequency but high-tokens-per-call (~3,000-7,500 prompt) and has the cleanest wins:

1. **Gate on a content-change hash (the big one).** It has **NO content gating** — enqueued after every non-media embed batch (`embed-job.ts:476-483`) with only a 60s debounce + 5-min min-interval (`stage-suggester-queue.ts:5-6`). Re-indexing identical KB re-runs the full-prompt call. Hash `(ordered stages JSON + loadKnowledgeSummary output)`, store on `stage_suggestion_jobs`, short-circuit when unchanged. **~60% of suggester cost.**
2. **Lower cadence.** Cron is `* * * * *` (`stage-suggestions-tick/route.ts:5`) for an **advisory feature a human reviews manually**. Move to hourly or fold into the existing daily `leads-dormant-sweep`. Skip when pending suggestions already cover every stage/field. ~25%.
3. **Shrink the prompt.** Drop offer/FAQ caps 50→~10 and truncation 240/200→~100 chars (`stage-suggester.ts:104-121`); sample diverse items (one per tag), not first-50. ~50% per-call.
4. **Cheaper model.** It uses `ragConfig.llmModel` (full chat model) for **offline advisory work** (`stage-suggester.ts:161`). Route to a cheaper tier — a human gates every suggestion, so quality risk is low.

For the **per-message auto-sort** (deep-reclassify / classifyOnly): same fixes as messaging item #11 — heuristic pre-filter on buying/disengage signals seeded from each stage's `entry_signals`, skip on won/lost terminal stages, reuse `conversation_summary` instead of 60 raw messages. ~40% of that path.

**Already optimal — leave alone:** the inline classifier (`answerWithClassification` stage_change) rides free on the reply call; `applyStageChange` (`classify.ts:471`) is a pure DB RPC; dormant-sweeper and suggestion-housekeeping are pure DB/heuristic (zero AI).

## 8. Scale projection

**Assumptions (stated):** 1 active client = 1 onboarded SMB, bot live. Blended Flash rate ~$0.10/M (90%+ input). Amortized **~$0.000475/inbound message** (main call ~4,050 tok + amortized deep-reclassify/summary ~684 tok). Sequences ~$0.00028 each. Onboarding ~$0.002 one-time (negligible). **Conservative** = 900 inbound + 150 sequences/mo/client. **Aggressive** = 6,000 inbound + 750 sequences/mo/client + recommend/rerank add-ons.

| Clients | Conservative BEFORE | Conservative AFTER (~55%↓) | Aggressive BEFORE | Aggressive AFTER (~55-65%↓) |
|---|---|---|---|---|
| **10** | ~$4.70/mo | ~$2.10/mo | ~$37/mo | ~$13-17/mo |
| **100** | ~$47/mo | ~$21/mo | ~$370/mo | ~$130-170/mo |
| **1000** | ~$470/mo | ~$210/mo | ~$3,700/mo | ~$1,300-1,665/mo |

Per-client/month: **~$0.47 (quiet) to ~$3.70 (busy) BEFORE**, **~$0.21 to ~$1.30-1.65 AFTER**. The savings concentrate in prompt caching + prose/context pruning on the per-message main call (the steepest curve — it multiplies by every message of every lead of every client).

**Note on non-token scaling risks** (not in the OpenRouter bill, but real at 1000 clients): five `* * * * *` crons + two `*/5` = ~1,440 ticks/day × 6 jobs × N clients of **serverless invocations** (Vercel/Supabase compute, not tokens — drains cost ~0 tokens on empty queues since they claim only `status='queued'`). And the **embed re-run hazard**: any blanket `embedding_status` reset fans out `enqueuePendingSources` (`embed-job.ts:97`) to one embed per chunk across the whole corpus — a one-shot spike that scales with KB size, not client count. Content-hash diffing (`ingest.ts:90`) prevents steady-state re-embeds; never blanket-reset status.

## 9. What NOT to change

| Item | Why it looks expensive but isn't / why changing it is a regression |
|---|---|
| **The combined reply+classify call mechanism** | Already merged into one envelope (`classify.ts:237`); classify is not a separate billed call. The lever is prompt size, not call count. |
| **The chat model (deepseek-v4-flash)** | ~$0.10/M is already a cheap tier. It writes persuasive Taglish sales copy + accurate product detail + reliable JSON. A regression here costs conversions worth far more than the token savings. Optimize input volume, not the model. |
| **`response_format=json_object`** | Cheapest structured-output mode. Switching to tool/function-calling ADDS tool-def tokens and reintroduces artifact leakage the code already strips (`classify.ts:857-920`). |
| **Cohere rerank** | Only ~$0.20 total, AND it does **not** fire on the production chat path (`retriever.ts:96` no-reranker branch — `classify.ts`/`answer.ts` pass no reranker). Lowest-priority lever. |
| **The embedding model (perplexity 0.6B, 1024-dim)** | Already a cheap floor. The lever is COUNT (caching/dedup), not unit price. DB stores 1024-dim vectors (`config.ts:30-31`) — swapping the model risks a **silent dimension-mismatch that corrupts retrieval**. If you must, re-embed the whole KB in lockstep and assert `length==1024`. |
| **Qualification scoring** | Rule-based, **zero AI** (`qualification.score.ts:95-166` pure arithmetic, `qualification-outcomes.ts:88-123` deterministic). Do NOT "AI-ify" it. |
| **CRAG query rewrite** | Well-gated to zero-candidate searches only (`retriever.ts:159`), bounded by 1,500ms. Negligible volume. Leave as-is. |
| **extractReminder** | Heuristic-gated by `hasTimeMarker` (`extract.ts:69-70`) — near-zero median cost. Fine. |
| **Rolling summary** | Well-gated (`shouldRollSummary`, fire-and-forget). At most widen interval 8→12 — low priority. |
| **Ingest dedup** | Already content-hash + chunk-diff safe (`ingest.ts:88-91`, `chunk-diff.ts:27-43`). Verification-only; add a unit assertion that a version-only bump yields zero re-embed. |
| **Crons claiming `status='queued'` only** | Empty-queue ticks cost ~0 tokens. The cron concern is platform compute, not OpenRouter spend. |
